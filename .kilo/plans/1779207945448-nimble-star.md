# Product Grid & Grid Layout Editing - Comprehensive Fix Plan

## Executive Summary

The product grid has **13 identified issues** causing malformed rendering across devices. The root causes fall into three categories: **column-count mismatches** between backend/frontend/edit-mode/mobile, **CSS conflicts** between aspect-ratio/min-height/containment, and **synchronization gaps** between JS viewport detection and CSS media queries.

---

## Issue Inventory

### ISSUE-1: Backend Validates 1-4 Columns, Mobile Uses 3 (HIGH)

**Files:** `backend/src/handlers/layouts.ts:78,331,438`, `frontend/src/components/layout/DraggableProductButton.tsx:230-234`

**Problem:** Backend validates `gridColumn` range as `1-4` for all layouts. On mobile (3 columns), a product saved at column 4 is clamped to column 3 in edit mode via `Math.min(position.gridColumn, GRID_COLUMNS_MOBILE)`. Two products at (col=3, row=1) and (col=4, row=1) would BOTH render at column 3, creating an overlap.

**Fix:**
- Option A: Make column count configurable per till, store it in the DB, and validate against it
- Option B (simpler): Keep 4-column storage but remap positions on mobile - redistribute column 4 items to the next row
- In `DraggableProductButton.tsx`, replace the simple `Math.min` clamp with a proper mobile remap: if `gridColumn > GRID_COLUMNS_MOBILE`, set `gridColumn = ((position.gridColumn - 1) % GRID_COLUMNS_MOBILE) + 1` and `gridRow = position.gridRow + Math.floor((position.gridColumn - 1) / GRID_COLUMNS_MOBILE)`

**Verify:** Place 4 products across row 1 (cols 1-4) on desktop, switch to mobile, confirm no overlaps.

### ISSUE-2: Normal Mode Mobile Drops All Positioning (HIGH)

**File:** `frontend/src/components/layout/DraggableProductButton.tsx:235-242`

**Problem:** In normal (non-edit) mode on mobile, the code intentionally removes all grid positioning:
```tsx
// Normal mode on mobile: no position, flows naturally
```
This means carefully arranged layouts are completely ignored on mobile devices during daily use. Products appear in DOM order instead of the user's custom arrangement.

**Fix:**
- Apply the same mobile remap logic (from ISSUE-1) in normal mode on mobile
- Remove the `else if (!isMobile)` guard so positions are always applied
- The only reason to drop positioning on mobile was to avoid column-4 overflow, which is fixed by ISSUE-1's remap

**Verify:** Arrange products on desktop, view on mobile in normal mode, confirm custom order is preserved.

### ISSUE-4: `aspect-ratio: 4/3` vs `min-height` Conflict (HIGH)

**File:** `frontend/src/index.css:248-249`

**Problem:** `.product-grid-button` sets both:
- `min-height: clamp(72px, 10vh, 120px)` (line 248)
- `aspect-ratio: 4 / 3` (line 249)

On a 320px screen with 3 columns + gaps (~100px per button), `aspect-ratio: 4/3` computes ~75px height. But `min-height` forces 72-120px. On medium screens (~150px per button), `aspect-ratio` computes ~112px while `min-height` might be ~100px (10vh). The taller value wins, creating inconsistent button proportions across screen sizes.

**Fix:**
- Remove `min-height` from `.product-grid-button` and rely solely on `aspect-ratio: 4/3`
- Or remove `aspect-ratio` and keep `min-height` with a responsive value
- Recommended: keep `aspect-ratio: 4/3` for consistent proportions, set a `min-height` only on `.product-grid-cell` (the drop zone) to ensure minimum drag target size

**Verify:** Check button proportions at 320px, 768px, 1024px, 1440px widths.

### ISSUE-5: `@container` Context Creates CSS Mismatch (MEDIUM)

**File:** `frontend/src/components/layout/ProductGridLayout.tsx:294`

**Problem:** The wrapping div has `@container` class, establishing a container query context. Child components like `AvailableProductsPanel` use container queries (`sm:grid-cols-3`, `md:grid-cols-4`). But the main grid (`product-grid-container`) uses viewport media queries (`@media (min-width: 768px)`). If the grid container is narrower than the viewport (sidebar layout, split screen), container queries and viewport media queries resolve differently, causing visual inconsistency between the grid and its child panels.

**Fix:**
- Decide on ONE strategy: either use container queries everywhere, or viewport queries everywhere
- Recommended: keep viewport queries for the main grid (simpler, already working), remove `@container` from the wrapping div in `ProductGridLayout.tsx:294`
- Keep `@container` only for components that need it (like `AvailableProductsPanel`)

**Verify:** Test at viewport 1024px with the POS sidebar visible. Check that grid columns match between the main grid and the available products panel.

### ISSUE-6: `!important` Inconsistency Across Grid Types (LOW)

**File:** `frontend/src/index.css:176-185,195-206,209-221`

**Problem:** `.product-grid-container` uses `!important` on `display`, `grid-template-columns`, `gap`, `width`, `min-height`. `.product-grid-unplaced` and `.product-grid-edit-overlay` do NOT use `!important`. This inconsistency means Tailwind utilities could accidentally override the unplaced and overlay grids but not the main container.

**Fix:**
- Either remove all `!important` from `.product-grid-container` (preferred - they're unnecessary since the CSS file loads after Tailwind)
- Or add `!important` to the other two grid classes for consistency

**Verify:** Confirm no Tailwind utilities override grid properties.

### ISSUE-7: Edit Overlay Grid Lines Don't Sync with CSS Grid (MEDIUM)

**File:** `frontend/src/components/layout/ProductGridLayout.tsx:309-317`

**Problem:** The edit mode visual grid lines use JavaScript-computed CSS gradient backgrounds:
```tsx
backgroundSize: `${100 / currentGridColumns}% 80px, ${100 / currentGridColumns}% 100%`
```
The JS `currentGridColumns` reads `window.innerWidth`, but the CSS grid switches columns via `@media (min-width: 768px)`. During resize, there's a brief window where JS has updated but the CSS media query hasn't triggered (or vice versa), causing misaligned grid lines.

**Fix:**
- Use CSS `repeating-linear-gradient` with `calc()` in the CSS file instead of inline JS-computed values
- Or use the CSS grid's own grid lines (via `grid-template-columns`) instead of background gradients
- Alternatively, use the same drop zone cells to provide the visual grid (they already do)

**Verify:** Slowly resize browser across the 768px breakpoint and check grid lines stay aligned.

### ISSUE-8: `contain: layout style paint` Clips Absolute Children (MEDIUM)

**File:** `frontend/src/index.css:256`

**Problem:** `.product-grid-button` has `contain: layout style paint`. This tells the browser the element's contents cannot affect anything outside its bounds. The absolute-positioned children (`.product-grid-overlay`, `.product-grid-fav`, `.product-grid-drag-handle`) use `inset: 0` or small offsets, which should be fine. But if any child extends beyond the box (e.g., a box-shadow or overflow text), it gets clipped.

**Fix:**
- Change `contain: layout style paint` to `contain: layout style` (remove `paint`)
- Or test visually to confirm no clipping occurs; if none, document this as intentional

**Verify:** Check that out-of-stock overlay, favourite star, and drag handle render fully on all devices.

### ISSUE-9: `will-change: transform` on All Buttons (LOW)

**File:** `frontend/src/index.css:257`

**Problem:** `will-change: transform` is applied to every `.product-grid-button` unconditionally. On grids with many products (50+), this promotes each element to its own GPU layer, consuming significant memory. This is particularly problematic on low-end mobile devices.

**Fix:**
- Remove `will-change: transform` from the base `.product-grid-button` rule
- Add it to `.product-grid-button.dragging` and optionally `.product-grid-button:hover`

**Verify:** Check for visual regression during drag operations.

### ISSUE-10: `content-visibility: auto` Scroll Jumping (LOW)

**File:** `frontend/src/index.css:376-381`

**Problem:**
```css
content-visibility: auto;
contain-intrinsic-size: 0 clamp(72px, 10vh, 120px);
```
The width `0` is incorrect for `contain-intrinsic-size`. When the browser estimates off-screen item sizes using this hint, it assumes width=0, causing the scrollbar to jump when items render. The estimated width should match the actual button width (roughly `1/3` or `1/4` of the grid width minus gaps).

**Fix:**
- Change to `contain-intrinsic-size: auto clamp(72px, 10vh, 120px)` to let the browser auto-detect
- Or use explicit values: `contain-intrinsic-size: 120px 100px` (approximate button width x height)
- Or remove `content-visibility: auto` entirely if scroll jumping is observed

**Verify:** Load a grid with 50+ products, scroll rapidly, check for scroll position jumps.

### ISSUE-11: Positioned + Unpositioned Products Create Chaotic Layout on Desktop Normal Mode (HIGH)

**File:** `frontend/src/components/layout/DraggableProductButton.tsx:222-243`

**Problem:** On desktop normal mode, products WITH saved positions get explicit `gridColumn`/`gridRow`, while products WITHOUT positions flow naturally (empty `gridStyle`). The naturally-flowing products fill gaps left by explicitly-positioned ones. For example, if product A is at (col=2, row=1), product B (unpositioned) flows into (col=1, row=1) - fine. But if product A is at (col=4, row=3), there may be empty cells at (col=1-3, row=3) that get filled by random unpositioned products, creating a disorganized appearance.

**Fix:**
- If ANY products in the current category have positions, ALL products should respect grid layout
- Unpositioned products should be placed after the last positioned row, flowing left-to-right
- Add logic in `ProductGridLayout.tsx` to assign temporary positions to unpositioned products based on the max occupied row/column

**Verify:** Create a partial layout (some products positioned, some not), confirm all products appear in a sensible arrangement.

### ISSUE-12: No Intermediate Breakpoint for Small Screens (LOW)

**Problem:** The grid jumps from 3 columns (below 768px) to 4 columns (768px+). On very small screens (320-400px), 3 columns with gap and padding gives each button only ~90-100px width. Product names at `clamp(1rem, ...)` may truncate aggressively.

**Fix:**
- Consider adding a 2-column breakpoint at ~400px:
```css
@media (max-width: 399px) {
  .product-grid-container { grid-template-columns: repeat(2, 1fr) !important; }
}
@media (min-width: 400px) and (max-width: 767px) {
  .product-grid-container { grid-template-columns: repeat(3, 1fr) !important; }
}
@media (min-width: 768px) {
  .product-grid-container { grid-template-columns: repeat(4, 1fr) !important; }
}
```
- Update `GRID_COLUMNS_MOBILE` and `MOBILE_BREAKPOINT` in `ProductGridLayout.tsx` and `DraggableProductButton.tsx` accordingly
- Add `GRID_COLUMNS_SMALL = 2` and `SMALL_BREAKPOINT = 400`

**Verify:** Test at 320px, 360px, 400px, 500px, 768px, 1024px.

### ISSUE-13: Duplicate `isMobile` Detection in Two Components (LOW)

**Files:** `frontend/src/components/layout/ProductGridLayout.tsx:105-112`, `frontend/src/components/layout/DraggableProductButton.tsx:166-173`

**Problem:** Both components independently detect mobile viewport using the same logic. If they ever go out of sync (different breakpoints or debounce), the grid cells and product buttons will disagree on column count.

**Fix:**

- Or move it into `LayoutContext` so it's a single source of truth

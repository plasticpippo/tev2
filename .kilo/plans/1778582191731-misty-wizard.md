# Plan: Add File Backup Option to "Create Backup" Button

## Problem
The "Create Backup" button only downloads a `.sql` database dump. Users need the option to also back up files (storage, uploads, config) in the same archive format already used by cloud backups.

## Current State
- **Frontend button** (`BackupSettings.tsx:734`) calls `POST /api/settings/backup` -> returns `.sql` file
- **Cloud backup** (`backup.ts:439`) already creates a full `tev2_full_*.tar.gz` with DB + `/app/storage/` + `/app/uploads/` + `.env` + `VERSION`
- **Restore** (`backup.ts:715`) already handles restoring from `.sql`, `.sql.gz`, and `.tar.gz` archives
- **Duplicate**: `POST /api/settings/backup` (settings.ts:512) and `POST /backup/local` (backup.ts:276) do the same DB-only dump

## Approach
Add a backup type selector to the frontend and a new full backup endpoint that reuses the cloud backup's archive logic but streams the result directly to the browser.

## Changes

### 1. Backend: New endpoint `POST /api/backup/local/full`
**File:** `backend/src/handlers/backup.ts`

Add a new route after the existing `POST /local` (line 276) that:
1. Runs `pg_dump` to a temp file (same as cloud backup, lines 470-484)
2. Creates staging dir with `database/`, `volumes/`, `config/` subdirs (lines 493-497)
3. Copies DB dump, tars `/app/storage/` and `/app/uploads/`, copies `.env` and `VERSION` (lines 499-506)
4. Writes `MANIFEST.txt` (lines 508-518)
5. Creates final `tar.gz` archive (line 521)
6. Streams the archive file as the response with `Content-Type: application/gzip` and `Content-Disposition: attachment`
7. Cleans up temp files in a `finally` block

This is synchronous (no job polling needed since there's no cloud upload). Timeout: 5 minutes to accommodate large file volumes.

**Note:** The existing `POST /local` (DB-only) remains unchanged.

### 2. Frontend: Backup type selector
**File:** `frontend/components/BackupSettings.tsx`

- Add state `localBackupType: 'database' | 'full'` (default `'database'`)
- Add two radio buttons or a select dropdown above the Create Backup button:
  - "Database Only" (default) - downloads `.sql`
  - "Full Backup" - downloads `.tar.gz`
- Modify `handleLocalBackup` (line 231):
  - If `database`: call existing `POST /api/settings/backup`, download as `.sql`
  - If `full`: call `POST /api/backup/local/full`, download as `.tar.gz`
- Update the description text to explain the two options

### 3. Translations
**Files:** `frontend/public/locales/en/admin.json`, `frontend/public/locales/it/admin.json`

Add new keys:
- `settings.backup.backupType.database` - "Database Only"
- `settings.backup.backupType.full` - "Full Backup (Database + Files)"
- `settings.backup.backupType.databaseDescription` - "Download the database as a SQL file"
- `settings.backup.backupType.fullDescription` - "Download a complete archive with database, uploaded files, and configuration"

### 4. Backend error translations (if needed)
**Files:** `backend/locales/en/*.json`, `backend/locales/it/*.json`

Add error key for full backup failure if not already covered by existing `errors.settings.backupFailed`.

## Files Modified
1. `backend/src/handlers/backup.ts` - New `/local/full` endpoint
2. `frontend/components/BackupSettings.tsx` - Backup type selector + handler update

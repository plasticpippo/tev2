# Plan: Restore from Uploaded Backup File

## Goal
Add a "Restore from File" section to Admin > Settings > Backup that lets the user upload a `.sql`, `.sql.gz`, or `.tar.gz` backup file and restore from it.

## Context

The backup page (`BackupSettings.tsx`) has 4 sections:
1. Local Backup (downloads `.sql` via `pg_dump`)
2. Cloud Backup Setup (MEGA)
3. Cloud Backup (create/schedule)
4. Cloud Restore (from MEGA)

This adds a new section **between 1 and 2**: **Restore from File**.

**Key infrastructure facts**:
- Backend container has `psql`, `tar`, `gunzip` (Alpine busybox)
- Docker volumes `storage_data` and `uploads_data` are mounted at `/app/storage` and `/app/uploads` respectively
- `cloud_backup_data` is mounted at `/app/backups`
- No Docker socket is mounted (and not needed for this feature)

**Supported restore operations by file type**:

| File Type | Database | Volumes | Config |
|-----------|----------|---------|--------|
| `.sql`    | Yes      | N/A     | N/A    |
| `.sql.gz` | Yes      | N/A     | N/A    |
| `.tar.gz` | Yes      | Yes     | Note only |

For `.tar.gz` archives: volumes are restored by extracting directly into `/app/storage` and `/app/uploads`. Config (`.env`, `VERSION`) cannot be modified from inside the container -- a note in the UI explains this.

## Steps

### Step 1: Backend -- Add `backupUpload` multer middleware

**File**: `backend/src/middleware/upload.ts`

Add a second exported multer instance:
```typescript
export const backupUpload = multer({
  storage: multer.memoryStorage(),
  fileFilter: (_req, file, cb) => {
    const ext = path.extname(file.originalname?.toLowerCase() || '');
    const validExts = ['.sql', '.gz', '.tar.gz'];
    // Check for .tar.gz, .sql.gz, or .gz extensions
    const isValid = validExts.some(e => file.originalname?.toLowerCase().endsWith(e))
                    || ext === '.sql';
    if (isValid) cb(null, true);
    else cb(new Error('Invalid file type. Only .sql, .sql.gz, and .tar.gz files are allowed.'));
  },
  limits: { fileSize: 500 * 1024 * 1024 }, // 500 MB
});
```

Import `path` at the top.

### Step 2: Backend -- Add `POST /api/backup/restore/upload` endpoint

**File**: `backend/src/handlers/backup.ts`

Add a new route after `POST /cloud/restore`. Import `backupUpload` from the upload middleware.

**Route definition**:
```typescript
backupRouter.post('/restore/upload', authenticateToken, requireAdmin, backupUpload.single('backup'), async (req, res) => { ... });
```

**Synchronous logic** (returns immediately):
1. Validate `req.file` exists
2. Validate file extension (`.sql`, `.sql.gz`, `.tar.gz`)
3. Create a restore job: `const job = createJob('restore')`
4. Start async restore in background (fire-and-forget with `.then()/.catch()`, same pattern as cloud restore)
5. Return `res.json({ jobId: job.id })`

**Async restore logic**:

```
For .sql files:
  1. Write buffer to /tmp/restore_{jobId}.sql
  2. Run: psql -h {host} -p {port} -U {user} -d {db} -f /tmp/restore_{jobId}.sql
  3. Clean up temp file

For .sql.gz files:
  1. Write buffer to /tmp/restore_{jobId}.sql.gz
  2. Run: gunzip -c /tmp/restore_{jobId}.sql.gz | psql -h {host} -p {port} -U {user} -d {db}
  3. Clean up temp file

For .tar.gz archives:
  1. Write buffer to /tmp/restore_{jobId}.tar.gz
  2. Create temp dir: /tmp/restore_{jobId}/
  3. Extract: tar xzf /tmp/restore_{jobId}.tar.gz -C /tmp/restore_{jobId}/
  4. Find DB dump: look in database/ subfolder for *.sql or *.sql.gz
  5. If .sql.gz, decompress
  6. Strip \restrict/\unrestrict lines from the SQL file (like restore-cloud.sh does)
  7. Drop + recreate database:
     a. psql -h {host} -p {port} -U {user} -d postgres -c "SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE datname = '{db}' AND pid <> pg_backend_pid();"
     b. psql ... -d postgres -c "DROP DATABASE IF EXISTS {db};"
     c. psql ... -d postgres -c "CREATE DATABASE {db} TEMPLATE template0;"
  8. Restore: psql -h {host} -p {port} -U {user} -d {db} < {clean_sql_file}
  9. Restore volumes (if present in archive):
     a. If volumes/storage_data.tar.gz exists: clear /app/storage/* and extract into /app/storage/
     b. If volumes/uploads_data.tar.gz exists: clear /app/uploads/* and extract into /app/uploads/
  10. Run ANALYZE: psql ... -d {db} -c "ANALYZE;"
  11. Clean up temp files
```

**Helper function**: Extract the DB connection parsing into a shared function (reuse from `POST /local` endpoint). Both endpoints parse `DATABASE_URL` the same way.

**Spawn pattern**: For each `psql`/`gunzip`/`tar` command, use `spawn` with:
- Array arguments (no shell injection)
- `PGPASSWORD` in env
- Stream stdout/stderr to job output via `updateJobStatus`
- Check exit code for success/failure

### Step 3: Frontend -- Add "Restore from File" section

**File**: `frontend/components/BackupSettings.tsx`

Add between Section 1 (Local Backup) and Section 2 (Cloud Backup Setup).

**State variables**:
```typescript
const [restoreFile, setRestoreFile] = useState<File | null>(null);
const [restoreFileConfirm, setRestoreFileConfirm] = useState('');
const [restoreFileJob, setRestoreFileJob] = useState<BackupJob | null>(null);
const [restoreFileLoading, setRestoreFileLoading] = useState(false);

```

**UI structure** (matching existing BackupSettings style):
```
Section: "Restore from File"
  Description: explains the feature and accepted formats
  Warning note: "This will overwrite the current database. For .tar.gz archives, volumes will also be restored."
  File input area:
    - Hidden <input type="file" accept=".sql,.sql.gz,.tar.gz">
    - Visible button "Select Backup File"
    - Shows selected filename + size after selection
  Confirmation:
    - Text: "Type CONFIRM to proceed"
    - Text input field
  Action buttons:
    - "Restore" (red, disabled until file + CONFIRM)
    - "Cancel" (grey, resets form)
  Error/success alerts
  Job progress (reuse renderJobProgress)
```

**Upload handler** (`handleRestoreFile`):
```typescript
1. Validate restoreFile and restoreFileConfirm === 'CONFIRM'
2. setRestoreFileLoading(true), clear error/success
3. const formData = new FormData()
4. formData.append('backup', restoreFile)
5. fetch('/api/backup/restore/upload', { method: 'POST', headers: getAuthHeaders(), credentials: 'include', body: formData })
   NOTE: Do NOT set Content-Type -- browser auto-sets multipart/form-data boundary
6. On success: get jobId, set initial job state, call pollJobStatus variant
7. On error: set error message
```

**Polling**: Reuse the existing `pollJobStatus` function with `'restore'` type but set state to `restoreFileJob`/`restoreFileError`/`restoreFileSuccess`. May need to add a parameter to `pollJobStatus` for custom state setters, or duplicate the polling logic inline (simpler, matches existing patterns in the component).

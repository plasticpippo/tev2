# Backup & Cloud Backup GUI Implementation Plan

## Summary

Expand the existing Backup tab in Settings to include cloud backup setup, scheduling, and restore functionality. The backend will expose new API routes that wrap the existing shell scripts' logic. The frontend will add UI sections for cloud configuration, scheduling, and restore.

**Key decisions:**
- Expand existing Backup tab (not a new page)
- MEGA credentials via form in settings (password not persisted, only used for login)
- Multi-provider design (abstract provider interface, MEGA implementation first)
- Full restore in GUI with database-only option
- GUI-configurable scheduling (enable/disable, set hour)
- Detailed confirmation dialog before restore

---

## 1. Shared Types

**File:** `shared/types.ts`

Add to the `Settings` interface:

```typescript
backup?: {
  cloud: {
    provider: 'mega' | string;       // extensible for future providers
    enabled: boolean;
    megaEmail: string | null;         // stored for display, NOT the password
    megaLoggedIn: boolean;            // runtime status, not persisted
    retention: number;                // 30 by default
  };
  schedule: {
    enabled: boolean;
    hour: number;                     // 0-23, default 4
    compress: boolean;                // compress before upload
  };
};
```

## 2. Backend API Routes

**File:** `backend/src/handlers/backup.ts` (new file, separate from settings handler)

Create a dedicated `backupRouter` with these endpoints:

### 2.1 Cloud Configuration

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/backup/cloud/status` | Check cloud backup status: MEGA installed? logged in? (email, storage info) |
| `POST` | `/api/backup/cloud/login` | Login to MEGA with `{ email, password }`. Password is used once, never stored. |
| `POST` | `/api/backup/cloud/logout` | Logout from MEGA |
| `POST` | `/api/backup/cloud/install` | Trigger MEGA CMD installation (runs setup script step 1) |

### 2.2 Backup Operations

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/api/backup/local` | Create local DB backup (existing behavior, moved from settings) |
| `POST` | `/api/backup/cloud` | Create cloud backup (DB + volumes + config -> MEGA). Returns job ID. |
| `GET` | `/api/backup/cloud/jobs/:id` | Poll backup job status (in-progress, success, failed) |

Cloud backup runs as async job (spawned child process running `backup.sh --cloud --compress`). The endpoint returns a job ID immediately. Frontend polls for completion.

### 2.3 Cloud Restore

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/backup/cloud/list` | List available cloud backups (name, size, date) |
| `POST` | `/api/backup/cloud/restore` | Restore from cloud. Body: `{ filename, dbOnly: boolean }`. Returns job ID. |
| `GET` | `/api/backup/cloud/jobs/:id` | Poll restore job status |

### 2.4 Schedule Management

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/backup/schedule` | Get current cron schedule status |
| `PUT` | `/api/backup/schedule` | Update schedule: `{ enabled, hour, compress }`. Creates/removes cron entry. |

### 2.5 Settings Integration

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/backup/settings` | Get backup settings (schedule + cloud config summary) |
| `PUT` | `/api/backup/settings` | Save backup settings |

All routes require `authenticateToken` + `requireAdmin`.

### 2.6 Implementation Details

- The backup handler spawns the existing shell scripts (`scripts/backup.sh --cloud`, `scripts/restore-cloud.sh`) as child processes rather than reimplementing their logic in Node.
- Job tracking uses a simple in-memory Map (no DB needed -- jobs are ephemeral).
- Each job has: `id`, `type` (backup|restore), `status` (pending|running|success|failed), `output` (log lines), `startedAt`, `completedAt`.

**File:** `backend/src/router.ts` -- Mount `backupRouter` at `/api/backup`.

## 3. Frontend: BackupSettings Component Rewrite

**File:** `frontend/components/BackupSettings.tsx` (rewrite)

The existing single-button component becomes a multi-section layout:

### Section 1: Local Backup (existing)
- "Create Backup" button -- downloads `.sql` file (unchanged behavior)
- Last backup timestamp display

### Section 2: Cloud Backup Setup
- **Status indicator**: MEGA installed? Logged in? Shows email if logged in.
- **Setup flow** (shown when MEGA not installed):
  - Button to install MEGA CMD (calls `/api/backup/cloud/install`)
- **Login form** (shown when MEGA installed but not logged in):
  - Email field
  - Password field (password type input)
  - Login button
- **Logged-in state** (shown when logged in):
  - Display: "Logged in as email@example.com"
  - Logout button
  - Cloud storage usage (from `mega-du`)

### Section 3: Cloud Backup
- **"Create Cloud Backup" button** (disabled if not logged in)
  - Shows progress via job polling
  - Displays success/failure with log excerpt
- **Schedule settings**:
  - Enable/disable toggle
  - Hour selector (0-23 dropdown)
  - Compress toggle
  - Save button

### Section 4: Cloud Restore
- **"Load Cloud Backups" button** (disabled if not logged in)
- **Backup list table**: filename, date, size, actions
- **Restore button** per backup
  - Opens confirmation dialog:
    - Shows backup details (filename, date)
    - Toggle: full restore vs database-only
    - Lists what will happen (drop DB, replace volumes, replace config)
    - "Type CONFIRM to proceed" input (must type "CONFIRM")
    - Cancel / Restore buttons
- **Restore progress**: polls job status, shows log output

### UI Style
- Follow existing dark theme (slate-800/900 + amber accent)
- Each section is a `bg-slate-800 p-4 rounded-md` card
- Use same button styles as existing components
- Use existing spinner SVG from the current BackupSettings

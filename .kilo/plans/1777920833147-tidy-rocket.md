# Tabs Feature Reliability Audit Report

## Scope

This audit covers the full tabs feature stack:
- **Backend**: `backend/src/handlers/tabs.ts` (REST API), `backend/prisma/schema.prisma` (Tab model)
- **Frontend State**: `frontend/contexts/TabManagementContext.tsx`, `frontend/contexts/OrderContext.tsx`, `frontend/contexts/UIStateContext.tsx`
- **Frontend Components**: `frontend/components/TabManager.tsx`, `frontend/components/TransferItemsModal.tsx`, `frontend/components/dashboard/UnifiedOpenTabs.tsx`
- **Services**: `frontend/services/transactionService.ts`, `frontend/services/orderService.ts`
- **Integration**: `frontend/contexts/PaymentContext.tsx`, `backend/src/handlers/transactions.ts` (process-payment)

---

## Findings

### CRITICAL Severity

#### C1. Non-Atomic Transfer Operation Can Cause Data Loss
- **File**: `frontend/contexts/TabManagementContext.tsx:125-181`, `frontend/services/transactionService.ts:119-124`
- **Root Cause**: `updateMultipleTabs` uses `Promise.all(saveTab(...), saveTab(...))` — two independent HTTP requests. If the source tab update succeeds but the destination tab update fails (network error, 409 conflict, server crash), items are removed from the source but never added to the destination. Items are permanently lost.
- **Impact**: Financial data loss. Items disappear from both tabs.
- **Reproduction**: Transfer items between tabs; kill the server after the first PUT completes but before the second.

#### C2. TOCTOU Race Condition in `handleAddToTab`
- **File**: `frontend/contexts/TabManagementContext.tsx:51-82`
- **Root Cause**: Tab items are read from `appData.tabs` (stale context state), mutated in-memory, then saved. If another client (or the same client via a background refresh) modifies the tab between the read and write, the second write overwrites the first.
- **Impact**: Items added by concurrent users can be silently lost.
- **Reproduction**: Two cashiers add items to the same tab simultaneously.

#### C3. Backend PUT Overwrites Items When `items` Field is Missing
- **File**: `backend/src/handlers/tabs.ts:253`
- **Root Cause**: `items: JSON.stringify(items || [])`. If `items` is `undefined` (not in request body), it evaluates to `[]`, wiping all existing items. The validation only guards against `items !== undefined && Array.isArray(items)` — if items is omitted entirely, the overwrite is silent.
- **Impact**: Accidental data loss if any caller sends a partial update without items.
- **Reproduction**: `PUT /api/tabs/1` with body `{"name": "renamed"}` → all items erased.

#### C4. No Database Unique Constraint on Tab Names
- **File**: `backend/prisma/schema.prisma:230-242`, `backend/src/handlers/tabs.ts:136-145`
- **Root Cause**: The `Tab` model has `name String` without `@unique`. The application-level duplicate check (`findFirst`) has a TOCTOU window — two concurrent POST requests with the same name can both pass the check and create duplicate tabs.
- **Impact**: Duplicate tabs with identical names, confusing POS operators and breaking tab-based workflows.

### HIGH Severity

#### H1. Non-Atomic Table Status Updates Across All Tab Operations
- **File**: `backend/src/handlers/tabs.ts:161-175` (create), `249-273` (update), `291-305` (delete)
- **Root Cause**: Tab CRUD operations and `updateTableStatus()` are separate database calls without a transaction. If the server crashes between them, the tab is created/updated/deleted but the table status is stale (e.g., table remains "occupied" after its last tab is deleted).
- **Impact**: Tables get stuck in wrong states, requiring manual intervention.
- **Note**: The process-payment handler correctly uses `prisma.$transaction` for atomic operations — the tab handlers should follow the same pattern.

#### H2. Table Status Transitions Not Validated
- **File**: `backend/src/handlers/tabs.ts:11-26`, `backend/src/utils/tableStatus.ts`
- **Root Cause**: `TABLE_STATUS_TRANSITIONS` is defined but never used by `updateTableStatus()`. Any status can be set from any current state. For example, a table in `AVAILABLE` state can be set to `BILL_REQUESTED` directly, which is invalid.
- **Impact**: Invalid state machine transitions corrupt table state.

#### H3. No Error Handling in Tab Management Context Handlers
- **File**: `frontend/contexts/TabManagementContext.tsx:39-49` (create), `51-82` (addToTab), `98-107` (saveTab), `109-114` (closeTab)
- **Root Cause**: `handleCreateTab`, `handleAddToTab`, `handleSaveTab`, and `handleCloseTab` have no try-catch. API failures (network errors, 409 conflicts, 403 forbidden) propagate as unhandled promise rejections. The UI shows no feedback — the modal closes or nothing happens.
- **Impact**: Users see silent failures with no indication of what went wrong. Duplicate name errors (409) are especially confusing since the user gets no message.

#### H4. `handleAddToTab` Updates Order Session After Tab Save, Then Clears Regardless
- **File**: `frontend/contexts/TabManagementContext.tsx:70-81`
- **Root Cause**: After saving the tab, the order session status is updated to 'assign-tab'. If that secondary call fails, `clearOrder(false)` still executes on line 80. The order is cleared but the session status is inconsistent.
- **Impact**: Order session state drifts from actual order state.

#### H5. `handleLoadTab` Replaces Current Order Without Confirmation or Save
- **File**: `frontend/contexts/TabManagementContext.tsx:84-96`
- **Root Cause**: When a user loads a tab, the current order items are replaced with the tab's items via `setOrderItems(correctedItems)`. If the current order has unsaved items (unlikely given auto-save, but possible during network issues), they are silently lost.
- **Impact**: Potential data loss if current order is unsaved.

#### H6. `updateTableStatus` Failures Are Silently Swallowed
- **File**: `backend/src/handlers/tabs.ts:11-26`
- **Root Cause**: `updateTableStatus` catches errors and returns `false`, but none of the callers (POST line 174, PUT lines 264/271) check the return value. The client receives a success response even though the table status update failed.
- **Impact**: Client believes operation succeeded, but table status is inconsistent.

#### H7. Payment Processing Doesn't Check for Other Active Tabs on Table
- **File**: `backend/src/handlers/transactions.ts:370-376`
- **Root Cause**: When paying with a tab, `process-payment` deletes the active tab and sets the table to `available`. It doesn't check if other tabs are still linked to the table (unlike the tab DELETE handler which uses `tableHasOtherTabs`).
- **Impact**: Table is set to "available" while it still has active tabs.

### MEDIUM Severity

#### M1. Payment Processing Silently Skips Tab Deletion on Error
- **File**: `backend/src/handlers/transactions.ts:364-368`
- **Root Cause**: `tx.tab.delete(...).catch(...)` — tab deletion failure is caught and logged but the transaction still succeeds. The tab becomes orphaned.
- **Impact**: Orphaned tabs accumulate in the database.

#### M2. `activeTab` State Typed as `any`
- **File**: `frontend/contexts/OrderContext.tsx:15-16, 32`
- **Root Cause**: `activeTab: any` loses all type safety. The comment says "Using any temporarily" but it's been in production.
- **Impact**: No compile-time checks for tab property access; refactoring is unsafe.

#### M3. Debug `console.log` Statements in Production Code
- **File**: `frontend/components/TransferItemsModal.tsx:63-96`, `frontend/services/transactionService.ts:69-90`
- **Root Cause**: Multiple `console.log` statements left in production code, including full tab data dumps.
- **Impact**: Performance overhead; potential information leakage in browser console.

#### M4. Case-Sensitive Duplicate Tab Name Check
- **File**: `backend/src/handlers/tabs.ts:137`
- **Root Cause**: `findFirst({ where: { name: name.trim() } })` is case-sensitive. "Tab 1" and "tab 1" are treated as different names.
- **Impact**: Operators create confusing duplicate tabs with different casing.

#### M5. Backend Variable Shadowing in PUT Handler
- **File**: `backend/src/handlers/tabs.ts:203, 245`
- **Root Cause**: `existingTab` is declared in two different scopes within the same function — first for duplicate name check, then for table assignment check. While technically different scopes, this is confusing and error-prone for maintainers.
- **Impact**: Maintenance risk; could lead to bugs during future modifications.

#### M6. Tab Deletion Requires Admin but Creation/Modification Does Not
- **File**: `backend/src/handlers/tabs.ts:285` vs lines 41/101/190
- **Root Cause**: DELETE uses `requireAdmin` but GET, POST, PUT only use `authenticateToken`. Any cashier can create, modify, or read any tab.
- **Impact**: Privilege escalation — cashiers can modify other users' tabs or reassign tables.

#### M7. Full Data Refetch After Every Tab Operation
- **File**: `frontend/services/transactionService.ts:91` (`notifyUpdates()`), `frontend/contexts/GlobalDataContext.tsx:103-117`
- **Root Cause**: Every `saveTab`, `deleteTab`, and `updateMultipleTabs` calls `notifyUpdates()`, which triggers `debouncedFetchData` — refetching ALL data (products, categories, users, tills, settings, transactions, tabs, stock items, stock adjustments, order logs, rooms, tables, tax rates). This is 13 API calls.
- **Impact**: Severe performance degradation on slow networks; unnecessary server load.

#### M8. `updateMultipleTabs` Triggers Double Data Refresh
- **File**: `frontend/services/transactionService.ts:119-124`
- **Root Cause**: Each `saveTab` call internally calls `notifyUpdates()`. `Promise.all` of two `saveTab` calls fires two `notifyUpdates`, and then `updateMultipleTabs` fires a third. Three data refreshes (39 API calls total) for a single transfer operation.
- **Impact**: Triple redundant network traffic; UI thrashing.

### LOW Severity

#### L1. No `role="dialog"` or `aria-modal` on Modals
- **File**: `frontend/components/TabManager.tsx:39`, `frontend/components/TransferItemsModal.tsx:114`
- **Root Cause**: Both modals render as `<div className="fixed inset-0...">` without `role="dialog"` or `aria-modal="true"`. Screen readers don't recognize them as dialogs.
- **Impact**: Accessibility violation (WCAG 4.1.2, 1.3.1).

#### L2. No Focus Trap or Escape Key Handling in TabManager
- **File**: `frontend/components/TabManager.tsx`
- **Root Cause**: Tab key navigates to elements behind the modal; pressing Escape does nothing. Only the X button closes the modal.
- **Impact**: Keyboard users and screen reader users can't easily dismiss the modal.

#### L3. No Focus Trap in TransferItemsModal
- **File**: `frontend/components/TransferItemsModal.tsx`
- **Root Cause**: Same as L2 — no focus trap. Has Enter key support for the new tab name input but no Escape key handling.
- **Impact**: Same as L2.

#### L4. Missing `aria-describedby` for Error Messages
- **File**: `frontend/components/TabManager.tsx:57-59, 70-74`
- **Root Cause**: The error message div has no `id` attribute, and the input has no `aria-describedby` pointing to it. Screen readers don't announce the error.
- **Impact**: Accessibility violation (WCAG 3.3.1).

#### L5. Sorting and Totals Recalculated on Every Render
- **File**: `frontend/components/TabManager.tsx:81-82`, `frontend/components/dashboard/UnifiedOpenTabs.tsx:17-20`
- **Root Cause**: `[...tabs].sort(...)` and `items.reduce(...)` run on every render without `useMemo`.
- **Impact**: Minor performance overhead; noticeable with many tabs/items.

#### L6. `handleCloseTab` Provides No User Feedback
- **File**: `frontend/contexts/TabManagementContext.tsx:109-114`
- **Root Cause**: The function silently deletes the tab. No toast notification, no modal update, no loading state.
- **Impact**: Users don't know if the operation succeeded or failed.

#### L7. TabManager Input Has No Enter Key Handler
- **File**: `frontend/components/TabManager.tsx:48-59`
- **Root Cause**: The tab name input has no `onKeyDown` or `onKeyPress` handler for Enter. Users must tap/click the "Create" button.
- **Impact**: Poor UX, especially on tablets.

#### L8. Missing Loading States During Async Operations
- **File**: `frontend/components/TabManager.tsx`, `frontend/components/TransferItemsModal.tsx`
- **Root Cause**: No loading/disabled state while `onCreateTab`, `onAddToTab`, `onConfirmMove` are in-flight. Users can double-click and trigger duplicate operations.
- **Impact**: Potential duplicate operations; no visual feedback during network requests.

## Remediation Plan (Prioritized)

### Phase 1: Critical Data Integrity Fixes

| # | Finding | Action | Effort |
|---|---------|--------|--------|
| 1 | C1 | Create a backend endpoint `PUT /api/tabs/transfer` that atomically moves items between two tabs within a single `prisma.$transaction`. Replace `updateMultipleTabs` with a single atomic call. | M |
| 2 | C2 | Implement optimistic locking on the backend: add a `version` integer column to the `Tab` model. The PUT handler should use `updateMany({ where: { id, version }, data: { ..., version: { increment: 1 } } })` and fail if version mismatched. The frontend should re-read and retry on conflict. | M |
| 3 | C3 | Fix the PUT handler: only overwrite `items` when explicitly provided in the request body. Change to `items: items !== undefined ? JSON.stringify(items) : undefined`. | S |
| 4 | C4 | Add `@@unique([name])` to the Tab model in the Prisma schema. Run a migration to add the database constraint. | S |

### Phase 2: Backend Atomicity and State Machine

| # | Finding | Action | Effort |
|---|---------|--------|--------|
| 5 | H1 | Wrap all tab CRUD + table status operations in `prisma.$transaction` blocks, matching the pattern used in `process-payment`. | M |
| 6 | H2 | Integrate `isValidStatusTransition()` into `updateTableStatus()`. Reject invalid transitions with a 409 error. | S |
| 7 | H6 | Check `updateTableStatus` return value in all callers. Roll back or return error if it fails. | S |
| 8 | H7 | Add `tableHasOtherTabs` check in the `process-payment` handler before setting table to "available". | S |

### Phase 3: Frontend Error Handling and UX

| # | Finding | Action | Effort |
|---|---------|--------|--------|
| 9 | H3 | Wrap all handler functions in `TabManagementContext` with try-catch. Show toast notifications on error. Surface duplicate name errors (409) to the user. | M |
| 10 | H4 | Move `clearOrder(false)` into a `.then()` chain that only fires after both the tab save AND session update succeed. | S |
| 11 | H5 | Add a confirmation dialog when loading a tab while the current order has items. | S |
| 12 | M2 | Replace `activeTab: any` with `activeTab: Tab \| null` in `OrderContext`. Fix all consumers. | S |
| 13 | M3 | Remove all `console.log` debug statements from production code. Use `logDebug` on backend, remove from frontend. | S |
| 14 | L6 | Add toast notifications for tab close success/failure. | S |
| 15 | L8 | Add loading/disabled states to prevent double-clicks during async operations. | M |

### Phase 4: Performance Optimization

| # | Finding | Action | Effort |
|---|---------|--------|--------|
| 16 | M7 | Replace `notifyUpdates()` with a targeted tab-only refresh mechanism. Add `refreshTabs()` that only fetches `/api/tabs`. | M |
| 17 | M8 | Remove `notifyUpdates()` from inside `saveTab`. Let callers control when to refresh. The atomic transfer endpoint from Phase 1 eliminates this for transfers. | S |
| 18 | L5 | Memoize sorted tab lists and tab totals with `useMemo` in `TabManager` and `UnifiedOpenTabs`. | S |

### Phase 5: Accessibility Compliance

| # | Finding | Action | Effort |
|---|---------|--------|--------|
| 19 | L1 | Add `role="dialog"`, `aria-modal="true"`, and `aria-labelledby` to both modal overlays. | S |
| 20 | L2-L3 | Implement focus trap (use a library or `@react-aria/focus`) and Escape key handling in both modals. | M |
| 21 | L4 | Add `aria-describedby` linking error messages to their associated inputs. | S |
| 22 | L7 | Add `onKeyDown` handler to the tab name input to trigger creation on Enter. | S |

### Phase 6: Authorization Hardening

| # | Finding | Action | Effort |
|---|---------|--------|--------|
| 23 | M6 | Define and enforce authorization rules: cashiers can create/modify their own tabs; admins can manage all tabs. Add ownership checks to the backend. | M |

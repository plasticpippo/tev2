# Fix Decimal Precision Issue: 6 Decimal Place Support

## Problem

After refactoring the system for 6dp cost support, the GUI still rejects values like `0.000001` with a validation error and restricts display/input to 2dp. Analysis reveals the refactoring is ~70% complete -- the database, migrations, backend cost functions, and `CostManagementPanel` displays are correct, but several gaps remain across frontend and backend.

---

## Plan

### Task 1: Convert UpdateCostModal input to VKeyboardInput

**File:** `frontend/components/CostManagementPanel.tsx` (lines 85-93)

The `UpdateCostModal` uses a plain `<input type="number">` while the rest of the POS uses `VKeyboardInput`. On this POS app's target devices (touch screens), the plain input relies on the native browser keyboard, which:
- May not provide a decimal separator depending on locale (Italian uses `,`)
- Has inconsistent `type="number"` validation across browsers
- Does not integrate with the app's virtual keyboard system

**Change:** Replace the plain `<input>` with `<VKeyboardInput k-type="numeric">`, add `placeholder` with the existing i18n key, and keep `step="0.000001"`.

```tsx
// Before (line 85-93):
<input
  type="number"
  step="0.000001"
  min="0"
  value={newCost}
  onChange={e => setNewCost(e.target.value)}
  className="w-full px-3 py-2 bg-slate-700 border border-slate-600 rounded text-white"
  required
/>

// After:
<VKeyboardInput
  k-type="numeric"
  type="number"
  step="0.000001"
  min="0"
  value={newCost}
  onChange={e => setNewCost(e.target.value)}
  placeholder={t('costManagement.newCostPlaceholder')}
  className="w-full px-3 py-2 bg-slate-700 border border-slate-600 rounded text-white"
  required
/>
```

Add import at the top of the file (after line 1):
```tsx
import { VKeyboardInput } from './VKeyboardInput';
```

**Verify:** Open the Update Cost Modal on the POS device, confirm virtual keyboard opens, type `0.000001`, submit successfully.

### Task 3: Fix ProfitAnalyticsPanel -- use formatCost for COGS displays

**File:** `frontend/components/ProfitAnalyticsPanel.tsx`

COGS (Cost of Goods Sold) values are derived from 6dp unit costs but displayed at 2dp via `formatCurrency`. This loses meaningful precision for low-cost items.

**Changes:**

1. **Line 4:** Update import to include `formatCost`:
   ```tsx
   // Before:
   import { formatCurrency } from '../utils/formatting';
   // After:
   import { formatCurrency, formatCost } from '../utils/formatting';
   ```

2. **Line 230:** COGS in summary cards:
   ```tsx
   // Before:
   {formatCurrency(summary.cogs)}
   // After:
   {formatCost(summary.cogs, 2)}
   ```
   Note: `formatCost(value, 2)` shows 2dp but is explicit about it. Alternatively use `formatCost(summary.cogs)` for full 6dp -- **this is a design decision** (see question below).

3. **Line 330:** COGS by category:
   ```tsx
   // Before:
   {formatCurrency(cat.cogs)}
   // After:
   {formatCost(cat.cogs, 2)}
   ```

4. **Line 377:** COGS by product:
   ```tsx
   // Before:
   {formatCurrency(p.cogs)}
   // After:
   {formatCost(p.cogs, 2)}
   ```

5. **Lines 215, 246:** Revenue and gross profit stay on `formatCurrency` (2dp) -- these are transaction-level display values.

**Design decision (confirmed):** COGS on the dashboard stays at 2dp via `formatCost(value, 2)`. Aggregated display values are intentionally 2dp since `Transaction.totalCost` is `Decimal(10,2)`. Using `formatCost` instead of `formatCurrency` is semantically correct (it's a cost, not a price) while keeping dashboard readability.

### Task 5: Fix backend margin calculation precision gap

**File:** `backend/src/services/costCalculationService.ts`

The `updateVariantTheoreticalCost` function (around lines 122-123) uses `subtractMoney` (2dp) to compute margin from a 6dp `theoreticalCost`. For very small costs, the cost gets rounded to 0 before subtraction, producing inaccurate margins.

**Change:**
```typescript
// Before:
const marginValue = subtractMoney(price, theoreticalCost);
const currentMargin = roundMoney(divideMoney(marginValue, price) * 100);

// After:
const marginValue = price - roundCost(theoreticalCost);
const currentMargin = roundMoney((marginValue / price) * 100);
```

Rationale:
- `subtractMoney` is 2dp via currency.js -- it would turn `3.00 - 0.000001` into `3.00` (cost disappears)
- Raw JS subtraction `3.00 - 0.000001` = `2.999999` which is precise enough for the margin percentage
- `roundCost(theoreticalCost)` ensures the cost is properly rounded at 6dp first
- `currentMargin` is stored at `Decimal(5,2)` so `roundMoney` (2dp) for the final value is appropriate
- Import `roundCost` is already present in the imports (line 3)

**Verify:** Set an ingredient cost to `0.000001`, recalculate variant cost, confirm margin is not `100%` but close to it (e.g., `99.99%`).

### Task 6: Add decimal validation to frontend cost input

**File:** `frontend/components/CostManagementPanel.tsx` (lines 49-55)

Currently the frontend only validates `isNaN(cost) || cost <= 0`. The backend enforces max 6dp, but the error message is generic. Add client-side max decimal validation for better UX:

```typescript
// After line 53 (the isNaN/cost <= 0 check):
const costStr = newCost.toString();
if (costStr.includes('.') && costStr.split('.')[1]?.length > 6) {
  setError(t('costManagement.costDecimalsLimit'));
  return;
}
```

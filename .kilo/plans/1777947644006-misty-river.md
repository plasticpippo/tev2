## CRITICAL Issues

### C1. Missing `useToast` import in `TabManagementContext.tsx`

**File:** `frontend/contexts/TabManagementContext.tsx:38`
**Impact:** Runtime `ReferenceError: useToast is not defined` when any tab error occurs. Tab operations silently crash instead of showing error feedback.

**Root Cause:** The `useToast` hook is called on line 38 but never imported. The file imports 10 modules but omits `import { useToast } from './ToastContext';`.

**Evidence:** Compare with `TableAssignmentContext.tsx:8` which correctly imports `useToast`.

**Fix:**
```
Add `import { useToast } from './ToastContext';` to the imports in TabManagementContext.tsx.
```

---

### C2. Race condition on table assignment — no transactional atomicity

**File:** `frontend/contexts/TableAssignmentContext.tsx:34-87`, `backend/src/handlers/tabs.ts:148-179`
**Impact:** Two cashiers can simultaneously assign the same available table. The frontend checks `table.status !== 'available'`, but both pass before either creates a tab.

**Root Cause:** Tab creation (POST /api/tabs) wraps tab creation + table status update in a transaction, BUT the table availability check happens on the frontend before the API call. There is no server-side "check-and-reserve" atomic operation.

**Contrast with Tabs:** The tabs endpoint uses `prisma.$transaction` for atomic operations. However, even it doesn't check table availability before assigning. The `validateAndUpdateTableStatus` validates transitions but doesn't prevent concurrent assignments.

**Fix:**
1. In `POST /api/tabs`, add a table status check within the transaction before creating the tab:
   ```typescript
   if (tableId) {
     const table = await tx.table.findUnique({ where: { id: tableId } });
     if (table.status !== 'available') {
       throw new Error('Table is not available');
     }
   }
   ```
2. In `PUT /api/tabs/:id`, same check when tableId changes.

---

### C3. `updateTablePosition` stale closure bug in `TableContext.tsx`

**File:** `frontend/components/TableContext.tsx:221-251`
**Impact:** During rapid drag operations, the rollback mechanism reads stale `tables` state from the closure, potentially reverting a table to an incorrect position.

**Root Cause:** The `updateTablePosition` callback depends on `tables` (line 251 dependency array). When called rapidly, it captures the `tables` snapshot from the last render, not the current state. The `originalTable` lookup at line 223 uses this stale data.

**Contrast with Tabs:** Tabs don't have drag-and-drop operations, so this pattern doesn't apply.

**Fix:** Use a ref for tables to avoid stale closure, or use the functional form of `setTables` to get the current state for rollback:
```typescript
const updateTablePosition = useCallback(async (id: string, x: number, y: number) => {
  setTables(prev => {
    const originalTable = prev.find(t => t.id === id);
    // Store original for rollback in a ref
    originalPosRef.current = { x: originalTable?.x, y: originalTable?.y };
    return prev.map(table => table.id === id ? { ...table, x, y } : table);
  });
  // ...then do the API call and rollback using the ref if needed
}, [addToast, t]); // Remove `tables` from deps
```

### C4. No optimistic locking on Table model — concurrent update data loss

**File:** `backend/prisma/schema.prisma:207-228` (Table model), `backend/src/handlers/tables.ts:219-235`
**Impact:** Two admin users editing the same table simultaneously will silently overwrite each other's changes. The last write wins.

**Root Cause:** The Tab model has a `version Int @default(0)` field with `updateMany({ where: { version } })` for optimistic locking (see `backend/src/handlers/tabs.ts:449-462`). The Table model has no version field, and updates use a plain `prisma.table.update()`.

**Fix:**
1. Add `version Int @default(0)` to the Table model in schema.prisma.
2. Create a Prisma migration: `npx prisma migrate dev --name add-table-version`
3. Update `PUT /api/tables/:id` to use `updateMany` with version check.
4. Update frontend to send/receive the version field.

## HIGH Issues

### H1. `bill_requested` status missing from frontend validation

**File:** `frontend/utils/validation.ts:140`
**Impact:** When editing a table in the admin panel, if the table's current status is `bill_requested`, the validation will reject it as "Invalid table status", preventing edits without first changing the status.

**Root Cause:** The `validateTableStatus` function lists only `['available', 'occupied', 'reserved', 'unavailable']`. The backend defines 5 statuses including `bill_requested` (`backend/src/utils/tableStatus.ts:6-12`).

**Fix:** Add `'bill_requested'` to the `validStatuses` array in `frontend/utils/validation.ts:140`.

---

### H2. Hardcoded English strings in `TableAssignmentContext.tsx`

**File:** `frontend/contexts/TableAssignmentContext.tsx:48-51,84-85,97`
**Impact:** Error messages appear in English regardless of the user's language setting. Breaks i18n compliance.

**Examples:**
- Line 48: `'This table is currently occupied...'`
- Line 49: `'This table is currently ${table.status.replace('_', ' ')}...'`
- Line 84: `'Failed to assign table. Please try again.'`
- Line 97: (console.error with no user-facing string)

**Contrast with Tabs:** `TabManagementContext.tsx` properly uses `t('tabs.createFailed')`, `t('tabs.addFailed')`, etc.

**Fix:** Replace all hardcoded strings with `t()` translation calls. Add corresponding keys to the i18n locale files.

---

### H3. Ctrl+R keyboard shortcut conflicts with browser reload

**File:** `frontend/components/TableManagement.tsx:521-528`
**Impact:** Pressing Ctrl+R in the Table Management admin page reloads the entire page instead of refreshing table data. The browser's native reload shortcut takes precedence.

**Root Cause:** The `useKeyboardShortcuts` hook registers `Ctrl+R` for `refreshData()`, but browsers intercept this shortcut before JavaScript can handle it.

**Fix:** Change the shortcut to `Ctrl+Shift+R` or `Alt+R` to avoid browser conflict.

---

### H4. Dual data fetching paths for tables creates stale data risk

**Files:** `frontend/components/TableContext.tsx:50-91` AND `frontend/contexts/GlobalDataContext.tsx:114-115`
**Impact:** Table data is maintained in two separate state trees:
1. `TableContext` — used by admin TableManagement component
2. `GlobalDataContext.appData.tables` — used by POS interface (TableAssignmentModal, etc.)

Changes made in admin don't reflect in POS until full app refresh, and vice versa. A table marked `occupied` in POS still shows `available` in admin until manual refresh.

**Root Cause:** The TableContext was designed as a standalone admin context, while GlobalDataContext serves the POS. Both independently fetch from `/api/tables`.

**Contrast with Tabs:** Tabs are fetched only in GlobalDataContext. TabManagementContext operates on `appData.tabs` from GlobalDataContext, ensuring a single source of truth.

**Fix:** Refactor TableContext to consume `appData.tables` from GlobalDataContext for read operations, using tableService only for write operations. Or consolidate table state entirely into GlobalDataContext.

---

### H5. Single loading boolean for all table operations

**File:** `frontend/components/TableContext.tsx:42`
**Impact:** Any table operation (fetch, add room, update table, delete, position update) sets the same `loading` flag, causing the full LoadingOverlay to appear for minor operations like drag-and-drop position saves.

**Root Cause:** The `loading` state is shared across all CRUD operations.

**Contrast with Tabs:** `TabManager.tsx` uses granular loading states: `isAdding`, `isLoading`, `isLoadingTab`, `isClosing` — one per operation type.

**Fix:** Replace the single `loading` boolean with operation-specific flags: `isFetching`, `isSavingRoom`, `isSavingTable`, `isDeletingRoom`, `isDeletingTable`, `isUpdatingPosition`.

---

### H6. `Tab` type missing `version` field — optimistic locking data not passed

**File:** `shared/types.ts:232-240`
**Impact:** The frontend `Tab` interface doesn't include the `version` field that the backend requires for optimistic locking. When `PUT /api/tabs/:id` is called, the version is undefined, causing the backend's version check to fail with a 400 error.

**Root Cause:** The schema has `version Int @default(0)` on the Tab model, and the backend handler checks `if (version === undefined || typeof version !== 'number' || version < 0)`. The `saveTab` function in `transactionService.ts` sends `tabData` as-is, which may or may not include version.

**Evidence:** The `handleSaveTab` in `TabManagementContext.tsx:110-126` spreads `activeTab` which is loaded from `appData.tabs`. If the backend returns `version` in the response but the frontend type doesn't declare it, TypeScript won't complain but the field still passes through.

**Fix:** Add `version?: number` to the `Tab` interface in `shared/types.ts:232-240`.

## MEDIUM Issues

### M1. TableContext file location inconsistency

**File:** `frontend/components/TableContext.tsx`
**Impact:** Code organization confusion. All other contexts are in `frontend/contexts/`. The TableContext is the only context in the `components/` directory.

**Fix:** Move `TableContext.tsx` from `frontend/components/` to `frontend/contexts/`. Update all imports.

---

### M2. `refreshData()` after save causes redundant network requests

**File:** `frontend/components/TableManagement.tsx:580-590`
**Impact:** After saving a room or table, `refreshData()` is called which fetches ALL rooms and ALL tables. But `addRoom`/`updateRoom` already update local state optimistically (lines 98-99, 117 in TableContext.tsx). This causes a visible flash and wasted bandwidth.

**Contrast with Tabs:** Tab operations in `TabManagementContext.tsx` rely on `notifyUpdates()` in the service layer, which triggers `GlobalDataContext`'s debounced refetch — a single consolidated refresh.

**Fix:** Remove explicit `refreshData()` calls in `handleRoomSave`/`handleTableSave`. The `notifyUpdates()` call in `tableService.ts` already triggers a debounced global data refresh.

---

### M3. TableLayoutEditor effect dependencies cause unnecessary re-renders

**File:** `frontend/components/TableLayoutEditor.tsx:111-127`
**Impact:** The `useEffect` that attaches mouse listeners depends on `handleMouseMove` and `handleMouseUp`, which are recreated on every render because they depend on `draggingTableId` and `localPositions`. This causes the effect to run on every render, removing and re-adding listeners constantly.

**Root Cause:** `handleMouseMove` (line 68) depends on `draggingTableId` (state). `handleMouseUp` (line 87) depends on `draggingTableId` and `localPositions` (state). Since these are in the useEffect dependency array, any state change during drag triggers listener reattachment.

**Fix:** Use refs for the mutable values that the handlers need, or use `useRef` for the handlers themselves to avoid the dependency cycle.

---

### M4. No error boundary for POS-side table interactions

**File:** N/A
**Impact:** The admin-side `TableManagement` is wrapped in `TableErrorBoundary`, but the POS-side components (`TableAssignmentModal`, `TabManager`) have no error boundary. A crash in the table assignment flow will crash the entire POS interface.

**Fix:** Create a lightweight error boundary wrapper for POS modals, or reuse `TableErrorBoundary`.

---

### M5. `updateTableStatus` service function exists but is only used in PaymentContext

**File:** `frontend/services/tableService.ts:148-169`
**Impact:** The `updateTableStatus` function is defined in `tableService.ts` and exported through `apiService.ts`, but it's only used in `PaymentContext.tsx`. The `TableAssignmentContext.tsx` relies on tab save operations to trigger status changes instead of calling `updateTableStatus` directly, creating an indirect and fragile coupling.

**Fix:** Use `updateTableStatus` directly when table status needs to change (e.g., in `handleTableAssign`), rather than relying on tab save side effects.

---

### M6. Inconsistent error handling patterns between table and tab contexts

**File:** `frontend/components/TableContext.tsx` vs `frontend/contexts/TabManagementContext.tsx`
**Impact:** TableContext wraps every operation in try/catch with toast + error state + console.error + throw. TabManagementContext only shows toast without setting error state. This means table errors appear both as toasts AND inline error banners, while tab errors only appear as toasts. Inconsistent UX.

**Fix:** Standardize on one pattern. The tab approach (toast-only) is cleaner for modal-based workflows. The table approach (toast + error state) is better for persistent views.

---

### M7. Room deletion bypasses backend protection

**File:** `frontend/components/TableContext.tsx:132-157`
**Impact:** The frontend `deleteRoom` callback filters tables locally (line 143: `setTables(prev => prev.filter(table => table.roomId !== id))`). But the backend `DELETE /api/rooms/:id` (rooms.ts:225-263) checks if the room has tables and returns a 400 error if it does. If the frontend somehow has stale data and tries to delete a room that was assigned tables between fetches, the backend blocks it but the frontend has already optimistically removed the tables from local state.

**Fix:** Don't optimistically remove tables on room delete. Wait for the backend success response, then call `refreshData()`.

### L3. `getStatusColor` duplicated across 3 components

**Files:** `TableLayoutEditor.tsx:130-139`, `TableAssignmentModal.tsx:100-109`, `TableManagement.tsx:615-628` (as `getTranslatedStatus`)
**Impact:** If a status color or label changes, it must be updated in 3 places.

**Fix:** Extract `getStatusColor`, `getStatusText`, and `getStatusLabel` into a shared utility like `frontend/utils/tableStatus.ts`.

### L4. `TableLayoutEditor` doesn't handle touch events for mobile

**File:** `frontend/components/TableLayoutEditor.tsx:52-127`
**Impact:** Table drag-and-drop in the layout editor only works with mouse events (`onMouseDown`, `mousemove`, `mouseup`). On tablets/touchscreens, the drag feature is completely non-functional.

**Fix:** Add touch event handlers (`onTouchStart`, `touchmove`, `touchend`) alongside mouse events.

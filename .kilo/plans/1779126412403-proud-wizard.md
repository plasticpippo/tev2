# Table System UI/UX Bug Fix Plan

## Summary
Found 9 issues across the table system: 4 functional bugs, 3 UX issues, 2 backend inconsistencies. The most impactful are the `loading`/`isFetching` mismatch (loading states never show) and the table assignment modal confirm button being pushed off-screen.

---

## 1. CRITICAL: `loading` vs `isFetching` property name mismatch

**Files:** `frontend/components/TableManagement.tsx:504, 752, 852, 978`
**File:** `frontend/components/TableContext.tsx:16,47` (exports `isFetching`)

The context exports `isFetching` but `TableManagement` destructures `loading`. Since `loading` doesn't exist on the context type, it's always `undefined` (falsy). This means:
- Line 752: Rooms loading skeleton never shows
- Line 852: Tables loading skeleton never shows  
- Line 978: `LoadingOverlay` component never renders

**Fix:** Change `loading` to `isFetching` on line 504 and all 3 usage sites.

---

## 2. CRITICAL: Table Assignment Modal confirm button pushed off-screen

**File:** `frontend/components/TableAssignmentModal.tsx:135,197,348`

The modal container has `max-h-[90vh]` and uses `flex flex-col`. The visual canvas section has `min-h-[400px]` (line 197). Combined with the room selector, table info section, action buttons, and status legend, the total height exceeds the viewport on many screens. The "Assign to Table" confirm button and Cancel button get pushed below the visible area with no scroll indicator.

**Fix:** Replace the `flex flex-col` layout with a scrollable body. Split the modal into:
- Fixed header (title + close button)
- Scrollable body (room selector + canvas + selected info)
- Fixed footer (action buttons)

Add `overflow-y-auto` to the body section and remove `min-h-[400px]` in favor of a responsive height like `max-h-[40vh] md:max-h-[50vh]`.

---

## 3. BUG: Missing `bill_requested` in admin status handling

**File:** `frontend/components/TableManagement.tsx:408-425,616-629`

Two related problems:
- **Status dropdown** (line 408-425): The `<select>` offers `available`, `occupied`, `reserved`, `unavailable` but NOT `bill_requested`. If a table is in `bill_requested` status, the dropdown shows a blank/empty value because the current value doesn't match any option.
- **`getTranslatedStatus`** (line 616-629): The switch statement has no `bill_requested` case, so it falls through to `default: return status` which shows the raw string `bill_requested` instead of a translated label.

**Fix:**
- Add `<option value="bill_requested">{t('tables.statusBillRequested')}</option>` to the dropdown (line 424)
- Add a `bill_requested` case to `getTranslatedStatus`
- Add the translation key `tables.statusBillRequested` to locale files

---

## 4. BUG: `reserved` and `bill_requested` share the same color

**Files:** 
- `frontend/components/TableLayoutEditor.tsx:134-135`
- `frontend/components/TableAssignmentModal.tsx:104-105`

Both statuses use `bg-yellow-500 border-yellow-600`. Users cannot distinguish between a reserved table and one where the bill has been requested.

**Fix:** Change `reserved` to a distinct color (e.g., `bg-blue-500 border-blue-600` or `bg-orange-500 border-orange-600`). Update in both files. Also update the `getStatusTextColor` in TableAssignmentModal.tsx.

---

## 5. UX: `reserved` missing from status legends

**Files:**
- `frontend/components/TableLayoutEditor.tsx:234-251`
- `frontend/components/TableAssignmentModal.tsx:380-400`

Both status legends show 4 statuses (available, occupied, bill_requested, unavailable) but omit `reserved`. After fix #4 gives `reserved` a distinct color, it must appear in the legend.

**Fix:** Add a `reserved` entry to both status legends with the color chosen in fix #4.

---

## 6. UX: `?` help badges are non-interactive spans

**File:** `frontend/components/TableManagement.tsx:695`

The `?` help badges use `<span>` with only a `title` attribute. They're not keyboard-accessible, not focusable, and not announced by screen readers. They look like clickable elements but aren't.

**Fix:** Either:
- (a) Convert to `<button>` with `aria-label` and a tooltip/on-click handler, or
- (b) Remove them entirely since the Quick Tips section already provides the same information

---

## 7. UX: Hardcoded English strings in empty states

**File:** `frontend/components/TableAssignmentModal.tsx:255,309,321`

Lines 255, 309, 321 contain hardcoded strings `"TABLE"` and `"ROOM"` instead of i18n translation keys.

**Fix:** Replace with `t('tableAssignmentModal.emptyTableIcon')` and `t('tableAssignmentModal.emptyRoomIcon')`, adding the keys to locale files.

---

## 8. BACKEND: Position update returns raw `items` without JSON parse

**File:** `backend/src/handlers/tables.ts:372`

`PUT /:id/position` returns `updatedTable` directly without running `safeJsonParse` on the `items` field, unlike `GET /` and `PUT /:id` which parse items. This creates API response inconsistency where `items` could be a string instead of an array.

**Fix:** Add the same `safeJsonParse` call used by other endpoints before returning the response.

---

## 9. BACKEND: Status update endpoint lacks optimistic locking

**File:** `backend/src/handlers/tables.ts:382-429`

`PUT /:id/status` uses `prisma.table.update()` directly without a version check, while `PUT /:id` (full update) and `PUT /:id/position` both use `prisma.table.updateMany({ where: { version } })` for optimistic locking. Under concurrent access, a status update could silently overwrite a newer version.

**Fix:** Add the same optimistic locking pattern used in the full update and position update endpoints.

---

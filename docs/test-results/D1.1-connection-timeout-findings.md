## 5. Retry Mechanisms Available

### Idempotency Key System (idempotency.ts)

**Format:** `{timestamp}-{uuid}-{itemsHash}`

```typescript
export function generateIdempotencyKey(items: Array<{ id: number | string; quantity: number }>): string {
  const timestamp = Date.now().toString(36);
  const uuid = crypto.randomUUID().split('-')[0];
  const itemsHash = items
    .map(i => `${i.id}:${i.quantity}`)
    .sort()
    .join('_')
    .split('')
    .reduce((hash, char) => ((hash << 5) - hash + char.charCodeAt(0)) | 0, 0)
    .toString(36);
  return `${timestamp}-${uuid}-${itemsHash}`;
}
```

### Backend Idempotency (transactions.ts:156-185)

```typescript
// Idempotency check - if key provided, check for existing transaction
if (idempotencyKey) {
  const expirationCutoff = new Date(Date.now() - IDEMPOTENCY_KEY_EXPIRATION_MS);
  const existingTransaction = await tx.transaction.findFirst({
    where: {
      idempotencyKey,
      userId, // Bind to user for security - prevent cross-user replay attacks
      idempotencyCreatedAt: { gte: expirationCutoff }
    }
  });
  
  if (existingTransaction) {
    // Return existing transaction - will be returned with HTTP 200
    return { isDuplicate: true, transaction: existingTransaction };
  }
}
```

**Retry Mechanism Features:**
- **Safe retry:** Same idempotency key returns existing transaction (HTTP 200)
- **24-hour expiration:** Idempotency keys expire after 24 hours
- **User binding:** Keys are bound to user ID to prevent cross-user replay attacks
- **Audit logging:** Idempotent replays are logged for audit purposes

## Conclusions

### Strengths
1. Well-implemented 10-second timeout with AbortController
2. Comprehensive idempotency system for safe retries
3. Order state preserved on failure
4. Atomic transaction processing ensures data consistency
5. User-friendly error messages with i18n support

### Potential Improvements
1. **No automatic retry** - User must manually click pay again
2. **No exponential backoff** - Retries use same timeout
3. **No connection status indicator** - Users not informed of connectivity issues proactively
4. **Payment API lacks timeout** - `processPayment` in transactionService.ts doesn't use `makeApiRequest` with timeout

---

## Recommendations

1. Add connection status indicator in UI
2. Implement automatic retry with exponential backoff for transient failures
3. Update `processPayment` to use `makeApiRequest` for consistent timeout handling
4. Consider adding explicit "Retry" button in payment modal for better UX
